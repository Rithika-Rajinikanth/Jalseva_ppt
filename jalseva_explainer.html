Step-by-Step Workflow

How JalSeva works,
from alert to closure.

Every alert follows a strict governed lifecycle. No shortcuts. No silent closures. The system runs 8 AI nodes, enforces SLA timers, and requires physical evidence before any complaint can be marked resolved.

01
Community ‚Üí System
Alert Submission
The Sarpanch or any registered user submits a water scarcity report via the API (or a future mobile app). They describe the issue, number of affected households, GPS location, and severity. The system instantly assigns an immutable Alert Code (e.g. ALERT-20240519-KOT-A3F1) and begins the governance pipeline.
Example: Lakshmi submits: "450 households in Kothapalli have no water for 5 days. Two borewells are dry. GPS: 17.9284¬∞N, 79.8945¬∞E. Severity: High."
‚Üí System responds in <2 seconds with a structured alert record, SLA timer started.
FastAPI POST /alerts/ JWT Auth PostgreSQL INSERT
02
AI Node 1 ‚Äî Machine Learning
ML Risk Scoring
The alert runs through the pre-trained water_governance_model.pkl (scikit-learn classifier trained on historical water crisis data). The model predicts a risk score (0‚Äì1) and recommended severity ‚Äî and can upgrade the severity if the ML model detects patterns the reporter may have under-stated.
Example: Lakshmi reports "High" severity. The ML model sees 450 households + borewell failure in a drought-prone district and outputs:
‚Üí ml_risk_score: 0.82 ¬∑ ml_predicted_severity: Critical ¬∑ confidence: 0.91
‚Üí Severity is automatically upgraded from High ‚Üí Critical. SLA drops from 24 hrs to 6 hrs.
scikit-learn joblib water_governance_model.pkl Rule-based fallback
03
AI Node 2 ‚Äî Generative AI
LLM Validation (Gemini 1.5 Flash)
Google Gemini reads the full alert context ‚Äî description, district, households, ML score ‚Äî and acts as a "senior government water governance analyst". It applies defined governance rules (100+ households = Approved, vague descriptions = Pending Review) and returns a structured validation decision with reasoning and urgency level.
Example Gemini output:
Status: Approved
Reason: Alert affects 450 households with confirmed borewell failures. ML risk score of 0.82 exceeds critical threshold. Immediate intervention required. This qualifies under Rule 1 (100+ households) and Rule 2 (ML risk >0.6).
Urgency: Immediate
LangChain + Gemini 1.5 Flash Structured Prompt Rule-based fallback
04
AI Node 3 ‚Äî Routing Engine
Smart Routing & SLA Assignment
Based on severity, risk score, and household count, the routing engine assigns the alert to the correct authority tier in India's 5-level governance hierarchy (Panchayat ‚Üí Block ‚Üí Mandal ‚Üí District ‚Üí State). It calculates the SLA window and sets a hard deadline in the database.
Example: ml_risk_score = 0.82, households = 450, severity = Critical
‚Üí Assigned to: STATE_OFFICER
‚Üí SLA: 6 hours (deadline: 23:14 same day)
‚Üí Priority Score: 4 / 4
‚Üí n8n fires email to State Water Board automatically
LangGraph routing_node SLA calculation n8n webhook
05
Officer Response
Action Submission + Contradiction Detection
The assigned officer submits their action (tanker deployment, borewell repair, etc.). When they submit a resolution claim (saying "problem is fixed"), the system runs the NLI Contradiction Detector ‚Äî a Facebook BART-large-mnli model that compares the original complaint against the resolution claim and flags suspicious mismatches.
Example Contradiction Detected:
Original: "Two borewells completely dry, 450 households affected for 5 days"
Resolution claim: "Routine maintenance performed. Issue resolved."
‚Üí contradiction_flag: true ¬∑ contradiction_score: 0.78
‚Üí Alert stays in PENDING_VERIFICATION, cannot close automatically.
facebook/bart-large-mnli Zero-shot NLI Ushahidi pattern
06
SLA Enforcement
Automatic Escalation if No Response
A Celery background worker runs every hour. It queries all open alerts past their SLA deadline and automatically escalates them to the next authority tier ‚Äî no human intervention needed. Each escalation fires the correct n8n webhook and is permanently logged in the audit trail.
Example: STATE_OFFICER has not responded in 6 hours.
‚Üí Celery detects breach at 23:15 ‚Üí Creates SLABreach record ‚Üí Escalates to Chief Secretary tier
‚Üí n8n fires escalate-state webhook ‚Üí Email to State Water Board CC: Chief Secretary office
‚Üí No one can pretend the alert doesn't exist.
Celery beat (hourly) Redis queue SLABreach table
07
Evidence Gate (FixMyStreet Pattern)
Independent Validation ‚Äî The Only Path to Closure
This is the most important feature. An alert cannot be closed by any officer or admin. The ONLY way to close an alert is for an independent VALIDATOR (an NGO, a field inspector, or a community representative) to submit a physical field visit report with findings, GPS evidence, and an explicit APPROVED decision. Gemini then scores the coherence of the validation against the resolution claim.
Example: WaterAid India's field officer visits on Feb 12.
‚Üí Files: GPS coordinates, borewell photo, water quality test result (pH 7.2, TDS 320), 15 household signatures
‚Üí Gemini coherence score: 0.89 / 1.0 (claim matches findings)
‚Üí contradiction_flag: false ‚Üí decision: APPROVED
‚Üí Alert status ‚Üí CLOSED. Audit trail permanently sealed.
VALIDATOR role (RBAC) Gemini coherence scoring Evidence vault (mWater pattern)
08
Immutable Record
Audit Trail ‚Äî Every Action Permanently Logged
Every state change ‚Äî creation, assignment, action, escalation, validation, closure ‚Äî is written to the audit_logs table with actor ID, role, timestamp, and before/after status. This log can never be edited. It creates a permanent, court-admissible chain of accountability.
The full timeline for ALERT-20240519-KOT-A3F1:
[17:14] CREATED by Sarpanch Lakshmi Devi (PANCHAYAT)
[17:14] ML scored 0.82 ‚Üí severity upgraded Critical (SYSTEM)
[17:15] VALIDATED + routed to STATE_OFFICER (SYSTEM/Gemini)
[23:15] SLA_BREACHED ‚Üí escalated to State Level (SYSTEM/Celery)
[Feb 10] ACTION_SUBMITTED by Rajesh Kumar (DISTRICT_OFFICER)
[Feb 12] VALIDATED + CLOSED by Priya Sharma (VALIDATOR/WaterAid)

PostgreSQL
The Single Source of Truth
Every alert, action, escalation, audit log and SLA breach record lives in PostgreSQL. Relations between tables enforce data integrity ‚Äî you cannot have a Validation without an Alert, an Action without an Actor.
Purpose: Permanent, relational, query-able governance record. Supports court-admissible audit trails. Enables dashboard analytics across all districts and states.
‚ú®
Google Gemini 1.5 Flash
AI Governance Analyst
Gemini reads alert descriptions in natural language and applies governance logic ‚Äî something rule engines can't do for free-text descriptions. It also scores how coherent a validator's findings are against the resolution claim.
Purpose: Replace expensive human review for initial triage. Detect vague or suspicious descriptions. Score evidence coherence to catch false closures.
üß†
LangGraph
The Governance Workflow Engine
LangGraph orchestrates the 8-node AI pipeline as a stateful directed graph. Each node can read and modify the alert state. Conditional edges route alerts to action or escalation based on outputs from earlier nodes.
Purpose: Turn a complex governance process into a deterministic, auditable, resumable AI workflow. Handles failures gracefully with rule-based fallbacks.
üîç
facebook/bart-large-mnli
Fraud Detection via NLI
Natural Language Inference ‚Äî given a premise (original complaint) and a hypothesis (resolution claim), the model classifies whether the claim contradicts, supports, or is neutral to the original. A contradiction score >0.5 flags the alert.
Purpose: Automatically detect false resolutions without manual review. The same pattern used by Ushahidi to flag disputed crisis reports.
‚öôÔ∏è
FastAPI + Celery + Redis
The Backend Infrastructure
FastAPI serves the governance API with async endpoints. Celery handles background tasks ‚Äî most importantly the hourly SLA breach checker that auto-escalates overdue alerts. Redis queues the tasks.
Purpose: SLA enforcement that runs even when nobody is watching. No alert can silently expire without system action.
üîó
n8n Webhooks
Notification Automation
7 distinct n8n webhook endpoints receive governance events (new alert, action, escalation, validation, status change) and handle email dispatch, data table updates, and routing logic ‚Äî all without changing the core API code.
Purpose: Every relevant person receives an email at every state change. Silence is not an option. Escalation emails go to increasingly senior officials automatically.
